To build a repository named "Enterprise System Design Lab" that serves as a comprehensive reference implementation of modern enterprise application architecture. The business domain should be a Supply Chain and Order Fulfillment Platform consisting of Customer Service, Product Service, Inventory Service, Cart Service, Order Service, Payment Service, Shipment Service, Notification Service, Search Service, and Analytics Service. The repository must be organized into independent, self-contained projects grouped by architecture and technology topics. Every project should start from a common baseline template service and be executable independently without dependency on other sections. The objective is to demonstrate architectural concepts, integration patterns, scalability techniques, cloud-native practices, and enterprise-grade engineering solutions rather than build a single evolving application.

The repository should begin with foundational projects covering CRUD operations, layered architecture, clean architecture, multi-module design, and REST API standards. It should then progress into database patterns using PostgreSQL, covering indexing, partitioning, transaction management, locking strategies, migrations, and optimization. Microservice-focused projects should demonstrate service decomposition, synchronous communication, centralized configuration, service discovery, API gateways, and inter-service communication. Messaging sections should separately cover RabbitMQ and Kafka. RabbitMQ projects must demonstrate work queues, direct exchanges, topic exchanges, fanout-based publish-subscribe communication, delayed processing, request-reply interactions, and dead-letter queues. Kafka projects must demonstrate producer-consumer patterns, consumer groups, partitioning, retry topics, dead-letter topics, idempotent consumers, Kafka Streams, transactional messaging, outbox patterns, and event sourcing.

The repository must include a dedicated Redis section demonstrating cache-aside, write-through, write-behind, distributed locking, session management, inventory caching, rate limiting, and Redis Pub/Sub. A NoSQL section should demonstrate MongoDB for flexible product catalogs and reviews, Cassandra for event stores and time-series data, DynamoDB for cloud-native persistence, and polyglot persistence strategies. A search section should use OpenSearch to implement indexing pipelines, product search, autocomplete, faceted search, relevance ranking, and Kafka-driven indexing. Resilience projects should demonstrate retries, circuit breakers, bulkheads, timeouts, fallbacks, and rate limiting using production-grade patterns.

Observability projects should implement centralized logging, metrics collection, distributed tracing, Prometheus, Grafana, and operational dashboards. Security projects should demonstrate JWT authentication, OAuth2, role-based authorization, API security, secret management, and identity-provider integration. Containerization projects should cover Docker image creation, Docker Compose orchestration, networking, volumes, and production image optimization. Kubernetes projects should demonstrate deployments, services, ingress controllers, configuration management, secret management, autoscaling, stateful workloads, and Helm packaging.

A dedicated AWS cloud-native section should demonstrate deployment and integration using ECS, EKS, RDS, ElastiCache, SQS, SNS, MSK, CloudWatch, X-Ray, S3, and serverless integrations. Advanced architecture projects should demonstrate Saga orchestration and choreography, CQRS, Event Sourcing, Outbox Pattern, Strangler Pattern, Backend-for-Frontend, Domain-Driven Design, Hexagonal Architecture, and Multi-Tenant Architecture. The final section should combine all major concepts into a production-ready platform including CI/CD pipelines, performance testing, scalability validation, disaster recovery strategies, chaos engineering experiments, monitoring, security hardening, and operational runbooks.

Each project must contain architecture diagrams, problem statements, solution descriptions, setup instructions, API documentation, infrastructure definitions, Docker support, automated tests, and detailed README documentation explaining the architectural objective, business scenario, design decisions, implementation approach, and operational considerations. The repository should be designed as a long-term architect reference platform, interview preparation resource, and hands-on enterprise technology laboratory.
